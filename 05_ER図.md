# ER図 ― つづけるリハ

本アプリはRDB（リレーショナルデータベース）ではなく、**localStorage上のJSONドキュメント**でデータを保持する。ここでは、その論理的なデータ構造を ER 図として表現する（外部キーや結合は概念上のものであり、DBによる制約はない）。

## 1. 全体像

データは大きく2系統。

- **設定系（現在のメニュー）**：`rehab-menu-v2` … セクションと項目。
- **記録系（日次）**：`rehab-data-v1` … 日付ごとに「その日のメニュー構成のスナップショット」と「達成状態」を保持。

各日付が自分のスナップショットを持つことで、メニュー変更後も過去日の達成率が変わらない。

## 2. ER図

```mermaid
erDiagram
  SECTION ||--o{ MENU_ITEM : "contains (sectionId)"
  DAY_RECORD ||--o{ DAY_ITEM_STATE : "has"
  DAY_RECORD ||--o{ SNAP_SECTION : "freezes"
  SNAP_SECTION ||--o{ SNAP_ITEM : "contains"
  SNAP_ITEM ||--|| DAY_ITEM_STATE : "checks for"
  MENU_ITEM ..> SNAP_ITEM : "copied as snapshot"

  SECTION {
    string section_id PK
    string title
  }
  MENU_ITEM {
    string item_id PK
    string section_id FK
    string title
    string icon
    string kind "slots | sets"
    json   labels "slots時の枠ラベル"
    int    sets "sets時のセット数"
    int    count "sets時の1セット量"
    string unit "回 | 秒"
    string note
    string timer_kind "none | single | interval"
    int    timer_sec
    int    rest_sec
  }
  DAY_RECORD {
    string date PK "YYYY-MM-DD"
  }
  SNAP_SECTION {
    string date FK
    string section_id
    string title
    int    order
  }
  SNAP_ITEM {
    string date FK
    string item_id
    string section_id
    string title
    string kind
    int    units "枠数 or セット数"
  }
  DAY_ITEM_STATE {
    string date FK
    string item_id
    json   checks "[true,false,...]"
  }
```

## 3. エンティティ説明

| エンティティ | 物理的な所在 | 説明 |
|---|---|---|
| SECTION | `rehab-menu-v2`.sections[] | 現在のメニューの見出し。 |
| MENU_ITEM | `rehab-menu-v2`.items[] | 現在のメニューの運動項目。`section_id` で SECTION に属する。 |
| DAY_RECORD | `rehab-data-v1`[date] | 1日分の記録。日付が主キー。 |
| SNAP_SECTION | DAY_RECORD.menu.sections[] | その日のセクション構成（凍結コピー）。 |
| SNAP_ITEM | DAY_RECORD.menu.items[] | その日の項目構成（凍結コピー）。 |
| DAY_ITEM_STATE | DAY_RECORD.state[item_id] | 項目ごとの達成配列（枠／セット単位の真偽）。 |

## 4. 関係（カーディナリティ）

| 関係 | 意味 |
|---|---|
| SECTION 1 ― 多 MENU_ITEM | 1つのセクションに複数項目が属する。 |
| DAY_RECORD 1 ― 多 SNAP_SECTION / SNAP_ITEM | 1日が、その日のメニュー構成一式を内包する。 |
| SNAP_SECTION 1 ― 多 SNAP_ITEM | その日のセクションに複数の項目スナップショットが属する。 |
| SNAP_ITEM 1 ― 1 DAY_ITEM_STATE | 項目スナップショットに対し達成状態が1対1で対応。 |
| MENU_ITEM ┄> SNAP_ITEM | 当日の同期時に現メニュー項目がスナップショットへコピーされる（参照ではなく複製。過去日は複製後に変更されない）。 |

## 5. 補足

- `DAY_RECORD` は実体としては `{ menu:{sections,items}, state:{itemId:[...]} }` という1つのJSON。ER図は理解のために構成要素を分解して示している。
- 「当日」のレコードのみ、起動・編集のたびに現在のメニューへ同期され、達成状態は引き継がれる。過去日のレコードは凍結され、達成率の計算は各日の `SNAP_ITEM` に基づく。
